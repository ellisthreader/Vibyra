# Decisions

## 2026-05-09: Language — Real i18n Layer (Manual Translation Tables, Hermes-Safe Date/Number Formatting)

Decision: Language now actually translates UI text and formats dates/numbers per language without depending on `Intl.DateTimeFormat`/`Intl.NumberFormat` (which silently fall back to en-US on most React Native Hermes builds because only en-US ICU data is bundled). New module `src/context/translations.ts` ships:
- A `TRANSLATIONS` map keyed by display language (`English`, `Español`, `Français`, `Deutsch`, `日本語`, `中文`, `Português`) with ~40 keys covering profile rows, section headers, billing & usage sheets, footer, ribbons, and CTAs.
- `translate(language, key)` with a fallback chain: language → English → key. Missing keys render the raw key (loud failure mode).
- `localizedDate(language, date)` using a per-language `MONTHS_LONG` array and per-language ordering (en `May 9, 2026`, fr `9 mai 2026`, de `9. Mai 2026`, ja/zh `2026年5月9日`, es/pt `9 de mayo de 2026`).
- `localizedNumber(language, value)` with per-language thousands separators (`","` for en/ja/zh, `"."` for es/de, `" "` for fr/pt).

`PreferencesContext` no longer exposes `Intl`-based formatters. It now exposes `t(key)`, `formatNumber(value)`, `formatDate(value)`, all backed by the manual implementations above. The `LANGUAGE_LOCALE` BCP-47 map was removed (no longer needed).

`ProfileSettingsGroup` (`chunk19.tsx`) gained an optional `key` field on each row. When present, `onSelect(key)` is called and `activeLabel` compares against `key` instead of `label` — this lets `ProfilePage.tsx` translate the visible `label` while keeping a stable English `key` for `PROFILE_ROW_TO_SHEET` routing in `profile/types.ts`. Without this split, translating the labels would have broken navigation.

Translated surfaces wired through `prefs.t(...)`:
- `ProfilePage.tsx` — three section titles, twelve row labels.
- `ProfileHero.tsx` — "tokens remaining", "Renews on" / "Current plan", "Free trial" / formatted date.
- `BillingSheet.tsx` — title, header tokens suffix, manage payments link, secure-payments footnote.
- `BillingCycleToggle.tsx` — Monthly / Annual / 2 MONTHS FREE chip.
- `BillingFeaturedPlan.tsx` — Recommended ribbon (auto), MOST POPULAR ribbon (mapped from `tier.badge.toLowerCase().includes("popular")`), Current chip.
- `UsageSheet.tsx` — title, kicker, hero footnote, 3 stat-tile labels, two section titles + "active" suffix, both empty-state messages, primary CTA (Upgrade plan / Manage plan).

File(s): `src/context/translations.ts` (new), `src/context/PreferencesContext.tsx` (drops `Intl`, uses manual formatters + `t`), `src/screens/workspace/inline/chunk19.tsx` (`ProfileSettingsGroup` row key/label split), `src/screens/workspace/inline/profile/ProfilePage.tsx`, `ProfileHero.tsx`, `BillingSheet.tsx`, `BillingCycleToggle.tsx`, `BillingFeaturedPlan.tsx`, `UsageSheet.tsx`.

Reason: User's previous "language doesn't work" was correct — only `Intl` formatters were wired, no UI strings translated, and even those formatters silently fell back to English on Hermes Android due to missing ICU data. Manual translation tables and manual date/number formatting guarantee a visible change on every supported language without a native build config change. Ships ~40 string keys across 7 languages = 280 entries — finite scope, all in one file.

How to apply: Add new translatable strings via a new key in every `Dict` in `translations.ts` (don't forget the seven languages — at minimum copy English to keep the fallback chain working, then translate). Use via `prefs.t("namespace.key")`. For routing/state lookups, never key state on a translated string — use a stable English key passed alongside the translated label (see `ProfileSettingsGroup`'s row `key`/`label` split). Do NOT reintroduce `Intl.DateTimeFormat` / `Intl.NumberFormat` for user-visible output — they will look correct in Expo Go (uses host runtime Intl) but silently fall back to en-US on real Hermes Android builds.

## 2026-05-09 (rev 3): Light Theme — Three-Path Transformer (Translucent Dark Surfaces, Light-Brand Contrast, Faint Tints)

Decision: The light-mode color transformer (`src/screens/workspace/styles/themeTransform.ts → transformColorForLight`) now has **three** decision paths instead of one neutral/branded split, fixing two visibility regressions:

1. **Branded colors path (chroma ≥ 0.18)**: no longer blindly kept. Compute WCAG relative luminance (`relativeLuminance` helper, sRGB-linear formula). If `luminance > 0.5` (poor contrast on white), darken to HSL `L = 0.42` preserving hue/saturation — fixes yellow `#FFE76A`/`#FFD166`/`#FFF200`, gold `#FACC15`, and bright green `#4ADE80` becoming illegible on white. If branded alpha is < 0.4 (faint tint), boost alpha to `min(0.4, alpha * 3)` so 8 %–20 % yellow fills retain visual presence on the now-white page (fixes the top-left token chip background). Brand colors with `lum ≤ 0.5` (neon purple `#C259FF`, blue `#7DA3FF`, red `#FF6B7E`, deep purple `#8B35FF`) stay untouched — they have natural contrast on white and are the brand secondary the user wants visible everywhere.

2. **Translucent dark neutrals path (chroma < 0.18, alpha < 1, l < 0.4)**: NOT inverted. Inverting `rgba(18,19,30,0.82)` to `rgba(near-white, 0.82)` over a white page collapses to white — invisible. These translucent-dark colors are always *elevated card surfaces*, so output a **solid off-white card** at HSL `L = 0.94`, saturation clamped to `0.18`, alpha `1.0`. Fixes community tabs (`communityTab`, `communityFilterButton`), `billingHeaderTokens`, `usageHeroCard`, `billingFooter`, and every `rgba(15,15,24,X)`/`rgba(21,22,34,X)` panel becoming a visible soft card on the white page.

3. **Other neutrals path**: standard HSL lightness inversion `l → 1 − l` preserving hue/saturation/alpha. Handles solid dark surfaces (`#02030C` page → near-white), white text (`#FFFFFF` → `#000000`), and white-overlay borders (`rgba(255,255,255,0.08)` → `rgba(0,0,0,0.08)` — subtle dark hairline on white).

Inline icon `color="#FFE76A"` on `TokenBalancePill` (`src/screens/workspace/inline/chunk3.tsx`) was bypassing the stylesheet, so it now goes through `useThemedColor("#FFE76A")` and darkens to readable amber in light mode.

File(s): `src/screens/workspace/styles/themeTransform.ts` (added `relativeLuminance`, restructured `transformColorForLight` with three paths), `src/screens/workspace/inline/chunk3.tsx` (`TokenBalancePill` flash icon themed).

Reason: User reported "buttons on community can't be seen well on light mode, same as tokens in top left." Root cause was rev 2's binary neutral/branded split: translucent dark surfaces inverted into invisible translucent-light surfaces, and light-luminance branded colors (yellow/green) stayed bright on white where they have poor contrast. WCAG luminance is the right perceptual metric, not HSL lightness.

How to apply: When adding a new branded color with high luminance (yellows, light cyans, light greens, light pinks), the transformer auto-darkens it for light mode — no special handling. When adding a new translucent dark elevated surface (`rgba(low-rgb, low-alpha)`), the transformer auto-converts to a solid off-white card. The two transformer thresholds (chroma `0.18`, luminance `0.5`, alpha gate `0.4`, dark-l gate `0.4`, output L `0.94`/`0.42`) are the only knobs — DO NOT add per-color overrides; if a specific surface looks wrong, prefer adjusting the source color slightly so it falls into a different path. For inline `<Ionicons color="..." />` colors (which bypass the stylesheet), wrap with `useThemedColor("...")`.

## 2026-05-09 (rev 2): Light Theme — Programmatic Color Inversion Across All 39 Stylesheets

Decision: Light mode now applies to **everything**, not just the SafeAreaView shell. The 39 stylesheet parts are no longer dark-only — at module load `src/screens/workspace/styles/index.ts` builds **two** StyleSheets: `darkSheet` (the original combined parts) and `lightSheet` (`darkSheet` run through `transformStyleMap` from `themeTransform.ts`). The exported `styles` is a `Proxy` that returns entries from whichever sheet is currently active. `setStylesScheme(scheme)` sets the module-level `activeScheme` variable; `App.tsx`'s `AppContent` calls it synchronously during render based on `usePreferences().effectiveScheme`, then re-renders the screen tree with `key={effectiveScheme}` so all children remount and re-read styles from the correct sheet.

`themeTransform.ts` rules:
- Each color literal (hex `#RRGGBB[AA]` or `rgb()`/`rgba()`) is parsed.
- **Chroma threshold 0.18** (max channel − min channel, on 0–1 scale) decides neutral vs. branded. `#02030C`, `#101219`, `#9C97AE`, `#FFFFFF`, `rgba(255,255,255,0.08)` etc. are neutral. `#C259FF`, `#7DA3FF`, `#FACC15`, `#4ADE80` etc. are branded and **kept untouched** so the neon-purple secondary and tier accents survive in light mode.
- For neutrals: convert to HSL, **invert lightness** (`l = 1 − l`), preserve hue/saturation/alpha. `#FFFFFF` (l=1) → `#000000`. `#02030C` (l≈0.027) → near-white. `rgba(255,255,255,0.08)` → `rgba(0,0,0,0.08)` so subtle white overlays become subtle black overlays.
- Only style keys that take colors are transformed (`color`, `backgroundColor`, `borderColor` family, `shadowColor`, `tintColor`, `placeholderTextColor`, etc.) — the transformer recurses into nested objects/arrays but ignores non-color values.

Inline `<Ionicons color="..." />` props bypass the stylesheet, so `useThemedColor(c)` was added to `PreferencesContext.tsx`. It returns the original color in dark mode and the transformed color in light mode. Profile-flow files updated to use it for previously-hardcoded near-whites: `ProfileHero.tsx` (pencil edit, calendar), `BillingFeaturedPlan.tsx` (Current chip checkmark), `BillingSheet.tsx` + `UsageSheet.tsx` (header back arrow), `SecuritySheet.tsx` (sign-out), `HelpSheet.tsx` (contact support). Inline pure-white icons that sit on saturated buttons (purple play, blue checkout, etc.) were left alone — those buttons stay branded in both modes so white-on-saturated stays correct.

File(s): `src/screens/workspace/styles/themeTransform.ts` (new — `transformColorForLight`, `transformStyleMap`, `themedColor`), `src/screens/workspace/styles/index.ts` (Proxy + `setStylesScheme` + dual sheets), `App.tsx` (calls `setStylesScheme` in render, `key={effectiveScheme}` on `WorkspaceScreen`/`AuthScreen`/`OnboardingScreen` to force remount), `src/context/PreferencesContext.tsx` (new `useThemedColor` hook), `src/screens/WorkspaceScreen.tsx` (band-aid inline shell/main overrides removed — transformer handles it), profile flow files updated for inline icon colors. Earlier inline `backgroundColor: "#F3F1F8"` overrides on `BillingSheet`/`UsageSheet` were removed (transformer covers them).

Reason: User's first round of "wire it up + status bar" was real but only flipped the SafeAreaView edges — `WorkspaceScreen`'s opaque `#02030C` `shell`/`main` Views and modal `billingScreen` `#07070C` painted over it, so the toggle appeared to do nothing inside the app. The user demanded "white still with the neon purple secondary applied to absolutely everything" — that requires either rewriting all 39 stylesheets by hand (hours of work, easy to miss spots) or programmatic transformation (chosen). The chroma-based neutral-vs-branded split keeps the brand identity intact across both modes without per-color manual mapping.

How to apply: New stylesheet entries that use only standard color forms (`#RRGGBB`, `rgba(...)`) get light variants automatically — no manual mapping needed. To add a new color that should NOT invert (e.g. a brand color with low chroma like a dark-only navy that you want to stay dark in light mode too), make it sufficiently saturated (chroma ≥ 0.18) or pick a different shade. New inline icon `color="#..."` props on themed surfaces should go through `useThemedColor(...)` — pure-white icons on always-saturated buttons don't need wrapping. Do not call `setStylesScheme` from anywhere other than `App.tsx`'s render — the `key={effectiveScheme}` remount is what makes children re-read the Proxy.

## 2026-05-09: Preferences Persistence — Appearance + Language Wired Up

Decision: Appearance (`dark` | `light` | `auto`) and language are now persisted preferences, not local UI state. A new `PreferencesProvider` (`src/context/PreferencesContext.tsx`) wraps `<AppProvider>` in `App.tsx` and:
- Persists `{ appearance, language }` to `localStorage` under key `vibyra.prefs.v1` (loaded synchronously on mount via `loadPrefs()`, mirrored on every change via an effect).
- Resolves `appearance === "auto"` via `useColorScheme()` from `react-native` and exposes `effectiveScheme: "dark" | "light"`.
- Maps `LANGUAGES` (`English/Español/Français/Deutsch/日本語/中文/Português` from `profile/types.ts`) to BCP-47 locales (`en-GB`, `es-ES`, `fr-FR`, `de-DE`, `ja-JP`, `zh-CN`, `pt-PT`) and exposes `formatNumber(value, options?)` and `formatDate(value, options?)` helpers built on `Intl.NumberFormat` / `Intl.DateTimeFormat`.

`App.tsx` reads `effectiveScheme` and applies it to the `SafeAreaView` background (`DARK_SHELL_BG = "#07070A"` vs `LIGHT_SHELL_BG = "#F3F1F8"`) and the `expo-status-bar` style (`light`/`dark`). The 39 inner stylesheets remain dark-only — full light-mode theming is deferred. The visible change today is: status-bar foreground colour, safe-area edges, and any surface that doesn't override `backgroundColor` shows through to the shell. `useProfileSheets` (`profile/useProfileSheets.ts`) no longer holds `appearance`/`language` itself — it pulls them from `usePreferences()` so all sheets read the same source of truth and writes survive reload.

Number formatting respects the chosen language: `BillingSheet` header tokens (`prefs.formatNumber(app.creditsBalance)`), `UsageSheet` hero values, allowances, and usage tile messages counts all go through `prefs.formatNumber`. e.g. `1,234,567` (en-GB) becomes `1.234.567` in de-DE — visible feedback that the language change worked. UI string translation is NOT implemented (no i18n library, no translation tables); strings stay English by design.

File(s): `src/context/PreferencesContext.tsx` (new), `App.tsx` (provider wrap, dynamic shell bg + StatusBar style), `src/screens/workspace/inline/profile/useProfileSheets.ts` (delegates to context), `src/screens/workspace/inline/profile/BillingSheet.tsx` + `UsageSheet.tsx` (use `prefs.formatNumber`).

Reason: User flagged "appearance does nothing" and "language doesn't work." Both were UI-only mockups in `useProfileSheets`'s local state — never persisted, never propagated. The agreed scope was: wire-up + persist + status bar for appearance; persist + locale-aware date/number formatting for language. Full light theme + full UI translation would be much larger jobs, called out and deferred.

How to apply: Add new locale-sensitive numeric/date displays via `prefs.formatNumber` / `prefs.formatDate` rather than `.toLocaleString()`. To extend language list, edit `LANGUAGES` in `profile/types.ts` AND add the new locale string to `LANGUAGE_LOCALE` in `PreferencesContext.tsx`. To make light mode actually look light, every `backgroundColor`, `borderColor`, and `color` literal in `src/screens/workspace/styles/part1`–`part39` would need theming via a colour-resolver hook keyed off `effectiveScheme` — do not start that piecemeal; design a palette and replace as a single sweep.

## 2026-05-08: Membership System — Hybrid IAP+Stripe, Plan Tiers, Real-Usage Credit Math, Daily Cap

Decision: Vibyra ships a four-tier membership system (Free £0, Starter £19, Builder £49, Pro £99 — monthly, with a `× 10` annual price = "2 months free" plus +10% credits/month for annual). Mobile uses Apple/Google IAP via existing `expo-iap` flow; web/desktop uses Stripe Checkout. Both paths converge on `App\Services\Billing\CreditDeductor` which is the single writer for `users.credits_balance`, `credits_used`, `daily_credits_used`, `daily_credits_reset_at`, and the new `credit_ledger` table (one row per spend, grant, refresh). Source of truth for plan/model/multiplier/topup data is `backend/config/billing.php`.

Credit math: `credits = ceil(openrouter_usd * 100 * effective_multiplier)`, minimum 1. `effective_multiplier` starts at the model's per-tier markup (`auto`/budget = 1.0, balanced = 1.15, premium = 1.35–1.5) and stacks `× 1.25` for long-context (≥100k tokens) and `× 1.20` for agent-mode (build prompts or skill mode=build). `ChatEndpoint::chat` does a pre-call estimate via `CreditCalculator::estimateCredits` (using fallback per-million-token pricing in `billing.fallback_pricing_per_million_usd`) to reject under-funded requests with `402`, then deducts the *actual* cost from OpenRouter's `usage` payload (`usage.cost`, `prompt_tokens`, `completion_tokens`) post-call. The OpenRouter request now includes `usage: { include: true }` so cost is returned. If `usage.cost` is missing, deduction falls back to the same fallback-pricing estimate. The flat `ChatModelMap::creditCost(modelKey)` (1/2/4/6) is no longer authoritative for chat — it remains in the trait only as a vestigial helper for unrelated agent endpoints.

Plan gating happens in three layers: (1) **model whitelist** — `CreditCalculator::planAllowsModel($plan, $modelKey)` returns false if the model's tier is not in the plan's `allowed_tiers`. ## 2026-05-08: `/api/chat` Hardened Against Abuse — Rate Limits, Size Caps, Locked Model Whitelist

Decision: `backend/app/Http/Controllers/Concerns/ChatEndpoint.php` now enforces three rate-limit buckets per request via `Illuminate\Support\Facades\RateLimiter`: per-user `12/min` (`chat:user:{id}:1m`), per-user `200/hour` (`chat:user:{id}:1h`), per-IP `30/min` (`chat:ip:sha1(ip):1m`). All three are hit on every request; any one tripping returns `429` with `retryAfter` (seconds). Limits live as class consts (`CHAT_PER_USER_PER_MINUTE`, `CHAT_PER_USER_PER_HOUR`, `CHAT_PER_IP_PER_MINUTE`). The endpoint also caps inputs at the controller boundary: prompt `<= 8000` chars (`CHAT_PROMPT_MAX_CHARS` → 413), `fileBody` `<= 20000` chars (`CHAT_FILE_BODY_MAX_CHARS` → 413), and `history` must be an array of `<= 20` items (`CHAT_HISTORY_MAX_ITEMS` → 422). Unknown `model` keys now silently coerce to `auto` instead of being passed through — `ChatModelMap::resolveOpenRouterModel` no longer has the `if (str_contains($model, '/')) return $model;` slash-bypass that previously let any client request any OpenRouter model regardless of the whitelist; new helper `isKnownModelKey` validates membership against `MODEL_MAP` before resolution.

Files: `backend/app/Http/Controllers/Concerns/ChatEndpoint.php` (new `enforceChatRateLimit`, size + history validation, model-key normalisation), `backend/app/Http/Controllers/Concerns/ChatModelMap.php` (removed slash bypass, added `isKnownModelKey`).

Reason: Prior state — only auth + credit balance gated the endpoint. With unbounded prompt size, OpenRouter input-token cost (which is per-token, not per-call) was effectively uncapped: `max_completion_tokens` only caps *output*. The slash-bypass meant a client could send `model: "anthropic/claude-opus-4-1"` while `creditCost("anthropic/claude-opus-4-1")` matched the cheap "mini" branch via `str_contains` heuristics — full premium cost on a 1-credit charge. No rate limiter at all meant a single token (or signup-spammed account) could fan out hundreds of requests per second. The OpenRouter key lives in `backend/.env` and is shared across all users, so a single abusive client drains the org's spend.

How to apply: When adding new OpenRouter-touching endpoints (e.g. agent streaming, future skills), mirror the same three-bucket rate-limit pattern and the same size caps. Never re-introduce a "raw model passthrough" — every model selection must be a key in `ChatModelMap::MODEL_MAP`. If you need to whitelist a new model, add the key to the map and the appropriate cost branch in `creditCost()`. Per-user credit checks remain the secondary defence; rate limits are the primary defence against burst abuse, and size caps are the primary defence against per-request token-cost abuse. Background callers should NOT bypass these limits by routing through a different endpoint — extend the limits, don't sidestep them.

## 2026-05-08: Shared Backend-Offline Gate For Background Requests

Decision: `src/utils/appApi.ts` exports a module-level `backendOfflineUntil` timestamp with `markBackendOffline()`, `markBackendOnline()`, `isBackendKnownOffline()` helpers and a 60s cooldown (`BACKEND_OFFLINE_COOLDOWN_MS`). `appApiRequest` accepts a fourth `meta?: AppApiRequestMeta` argument with `{ background?: boolean }`. When `background: true` and the gate is set, the request is short-circuited via a `BackendOfflineError` BEFORE the fetch is even attempted — so the browser never logs `ERR_CONNECTION_REFUSED` for skipped background calls. Network errors and 5xx responses set the gate; successful responses clear it. Foreground (user-initiated) requests still attempt regardless and let the user see the failure.

Files: `src/utils/appApi.ts` (gate state, helpers, `BackendOfflineError`, updated `appApiRequest` signature), `src/context/useCloudSync.ts` (passes `{ background: true }` to `/api/session/state`), `src/context/AppContext.tsx` (passes `{ background: true }` to `/api/skills`).

Reason: The previous per-effect cooldown in `useCloudSync` only suppressed *its own* repeats; every other background caller (skills fetch, future polls) emitted its own `ERR_CONNECTION_REFUSED` log when the backend was down, and any state change after the 30s window ticked over would log another. With multiple callers and the user typing, a single backend outage produced repeated console noise. Centralising the gate means: one log on first failure, then total silence for 60s across all background callers, then one probe attempt — repeat. User-initiated requests (chat, agent start, login) bypass the gate so failures still surface to the user.

How to apply: Any new background poll/sync call to the Laravel backend MUST pass `{ background: true }` as the fourth `appApiRequest` argument. User-initiated requests must NOT pass `background: true` — they need to fail loudly so the user understands the action didn't go through. The browser's native `ERR_CONNECTION_REFUSED` log cannot be suppressed once a request fires; the only way to silence it is to skip the request, which is what the gate does. If a foreground request needs the gate (rare — e.g. a "preflight" liveness probe), do it explicitly via `isBackendKnownOffline()` rather than weakening the foreground/background distinction.

How to apply: Any new chat-rendering code must read from `chatThreads[<projectIdFromSelectedChatId>]`. Writes from `useAgentActions.setChatMessages` still go through `selectedProjectId` — the auto-snap effect in `src/screens/workspace/hooks/useWorkspaceActions.ts:36-40` keeps the two ids aligned after commit, so writes land in the visible thread in steady state.

## 2026-05-07: Initialize Fresh Chat Thread On Project Creation

Decision: When a project is created (desktop path or local fallback) `useWorkspaceActions.createProject` and `createLocalProject` explicitly write `chatThreads[newId] = []` and `chatTitles[newId] = name` via `setters.setChatThreads` / `setters.setChatTitles`.

Files: `src/context/useWorkspaceActions.ts` — `createProject` (desktop success path) and `createLocalProject`.

Reason: Defensive guard. Even though desktop project ids are unique (base64 of disk path, paths get `-2`/`-3` suffixes in `backend/app/Services/Concerns/ProjectFileState.php::createProject`), an id could be re-used after delete-then-recreate, and persistence layers could carry a stale entry. Initializing the bucket guarantees a freshly-made project always opens to an empty chat.

How to apply: Any new project-creation code path must zero `chatThreads[id]` and set a fresh `chatTitles[id]`. Skipping this can resurrect old conversations in newly-created projects.

## 2026-05-07: Cap OpenRouter `max_completion_tokens` Per Request

Decision: `ChatEndpoint::chat` always sends `max_completion_tokens` — `800` for chat, `3000` when `isBuildPrompt()` matches.

Reason: Without it OpenRouter reserves the model's full output window against the user's credit balance, which fails small balances with "requested up to N tokens" errors. Capping output also bounds worst-case cost per turn.

How to apply: Any new OpenRouter call site (streaming or not) must set `max_completion_tokens`. Build-mode budget is the only reason to raise it.

## 2026-05-07: Slim System Prompt By Default

Decision: `ChatPrompting::systemPrompt` returns a one-line system prompt for plain chat, and only ships the runnable-app HTML/CSP instructions when `isBuildPrompt()` matches.

Reason: The full build-mode prompt is ~400 tokens of input on every turn, even for "what does X mean" questions. Detect intent before sending it.

How to apply: Keep `isBuildPrompt()` as the single source of truth for build mode (used by system prompt selection, history sizing, and `max_completion_tokens`). Mirror it in the frontend (`useAgentActions.ts`) so payloads agree.

## 2026-05-07: Cloud Sync Backs Off On Failure

Decision: `useCloudSync` skips the debounced `POST /api/session/state` for 30s after any failure, and only emits the "Saved locally" log once per outage.

Reason: Every state change re-fires the effect; with the backend down this produced an `ERR_CONNECTION_REFUSED` per keystroke in the browser console. The native log can't be suppressed — only the request can.

How to apply: When adding new background syncs, gate them with the same cooldown pattern. User-initiated requests (chat, agent start) must NOT use the cooldown — they need to fail loudly.
